# ProhoriPay — Data Simulation

All data in ProhoriPay is synthetic. No real MFS account, wallet, balance, PIN, OTP, or credential exists anywhere in the system.

---

## Reproducibility

The generator is fully deterministic:

- **Fixed seed:** `numpy.random.RandomState(seed=1)` and `Faker.seed(1)` are set before any value is drawn.
- **Fixed reference clock:** `REFERENCE_NOW = datetime(2026, 7, 11, 12, 0, 0)`. The generator never reads the wall clock. All transaction timestamps are integer-second offsets from midnight of the reference day (`_DAY_START = datetime(2026, 7, 11, 0, 0, 0)`).
- **Consequence:** `python -m app.core.seed` always produces the identical agent, pools, transactions, and anomaly labels. Measurements in [`docs/validation.md`](validation.md) are reproducible by anyone running the seed command.

Source: `backend/app/modules/synth/config.py` (constants), `backend/app/modules/synth/generator.py` (logic).

## What is Generated

### One Super Agent

```
id:       AGENT_07
name:     Karim Store
area:     Sylhet-Zindabazar
providers: [bkash, nagad, rocket]
```

### Four Balance Pools (the core domain structure)

| Pool ID | Kind | Opening balance (BDT) | Notes |
|---|---|---|---|
| `physical_cash` | Shared physical cash | **Derived** (see below) | The sole constraining pool |
| `bkash` | Provider e-money | 2,500,000 | Separate, non-interchangeable |
| `nagad` | Provider e-money | 1,200,000 | Separate, non-interchangeable |
| `rocket` | Provider e-money | 1,800,000 | Separate, non-interchangeable |

**Why physical cash opening is derived:** The generator pins the *current* physical cash to `physical_target_current = 80,000 BDT` and works backward: `opening = target_current − net_physical_effects`. This ensures the domain invariant `current == opening + Σ(effects)` holds exactly for every pool, while guaranteeing physical cash is always the constraining pool regardless of the transaction mix. The provider e-money openings (totalling 5,500,000 BDT) make the combined total look healthy while physical cash sits at only 80,000 BDT — this is the hidden-shortage scenario.

## Direction-Aware Signed Pool Effects

Every transaction carries `pool_effects`: a list of `{"pool_id": str, "delta": int}` pairs, built by `build_pool_effects(txn_type, provider, amount)` in `backend/app/core/effects.py`.

| Transaction type | Physical cash delta | Provider e-money delta |
|---|---|---|
| `cash_out` | `−amount` | `+amount` (provider earns the float) |
| `cash_in` | `+amount` | `−amount` (provider float is disbursed) |

**No service or query reduces a balance without flowing through these signed effects.** The net of all effects over the entire transaction history equals the difference between opening and current balance for every pool — this is verified in integration tests. Provider e-money and physical cash drain in *opposite directions*, which is the domain rule that makes naive combined totals misleading.

## Temporal Seasonality

### Hour-of-day profile

The off-hours anomaly detector uses an intraday multiplier table that mirrors the expected traffic profile. This same table defines what "normal" means for a given hour:

```python
# backend/app/modules/alerts/config.py — hour_multipliers
{
 0: 0.03,  1: 0.02,  2: 0.02,  3: 0.02,  4: 0.02,  5: 0.05,
 6: 0.15,  7: 0.40,  8: 0.70,  9: 0.90, 10: 1.00, 11: 1.00,
12: 0.90, 13: 0.85, 14: 0.80, 15: 0.75, 16: 0.70, 17: 0.65,
18: 0.60, 19: 0.50, 20: 0.35, 21: 0.25, 22: 0.15, 23: 0.08,
}
```

Peak hours (10:00–11:00, multiplier 1.0). Deep off-hours (00:00–05:00, multipliers 0.02–0.05). The base expected rate at peak is `off_hours_base_rate = 0.20 txn/min per provider`.

### Seeded transaction mix (the 3-hour history)

Two phases of normal traffic, generated by `_generate_normal`:

| Phase | Window | Count | Character |
|---|---|---|---|
| Baseline | 09:00–10:00 | 26 | Balanced: 50% cash_out, 50% cash_in. Cash-out 500–5,000 BDT; cash-in 1,000–12,000 BDT. |
| Eid rush | 10:00–12:00 | 70 | Cash-out heavy: 82% cash_out, 18% cash_in. Cash-out 1,500–9,000 BDT. |

Provider distribution for cash-out (from `NormalTrafficSpec.provider_weights`):

```python
{"bkash": 0.45, "rocket": 0.35, "nagad": 0.20}
```

**Eid transaction mix:** High cash-out share (0.82) during the Eid window means physical cash drains fast while provider e-money float accumulates — exactly the hidden-shortage scenario. The off-peak baseline (balanced mix) is used by the velocity detector as the "quiet pace" from which Eid-rush rates are compared.

### Salary-day calendar

The event calendar declares a `salary_day` window (07:00–09:00) even though no seeded salary traffic exists. This demonstrates the detection architecture is ready for salary-day patterns without touching detection logic. During a real salary day, cash-in volume would be high (customers depositing salary disbursements) — draining provider float while filling physical cash (the reverse of Eid pressure).

## Injected Labeled Anomalies

Three clusters are injected after the normal traffic is generated, by `_generate_anomalies`. All are `cash_out`, all are tagged `is_injected_anomaly=True`:

| # | Type | Provider | Count | Accounts | Amount range (BDT) | Window | Event flag |
|---|---|---|---|---|---|---|---|
| A | `structuring` | bKash | 12 | ACC_9001, ACC_9002, ACC_9003 | 4,910–4,990 | 11:00–11:45 (inside Eid rush) | `eid_rush` |
| B | `velocity_spike` | Rocket | 18 | ACC_9101, ACC_9102 | 1,000–3,000 | 11:30–11:36 (6-min burst) | `eid_rush` |
| C | `off_hours_burst` | Nagad | 9 | ACC_9201, ACC_9202 | 2,000–4,000 | 03:05–03:35 (off-hours) | none |

**Total injected anomaly transactions: 39** (`EXPECTED_ANOMALY_COUNT = 39`). This is the ground truth the anomaly tests assert against.

Design intent:
- **Cluster A (structuring):** Near-identical amounts (~4,950 BDT ≈ just below the 5,000 BDT regulatory per-transaction threshold) from only 3 accounts in 45 minutes. The structuring detector finds this via amount clustering + account concentration, *not* raw volume — so the surrounding 70 Eid transactions (varied amounts, many accounts) do not trigger it.
- **Cluster B (velocity spike):** 18 transactions in 6 minutes from 2 accounts on Rocket. Rate ≫ the Eid-adjusted baseline. Tests whether the Eid multiplier still catches this extreme burst.
- **Cluster C (off-hours burst):** 9 transactions at 03:05–03:35 on Nagad. Hour-03 multiplier is 0.02 → expected rate is 0.004 txn/min; observed is far higher. No event calendar explains it. Tests the time-of-day profiling path.

## Where the Data Lives

**Database:** SQLite at `backend/prohoripay.db` (path from `DATABASE_URL=sqlite:///./prohoripay.db`).

**Seed entrypoint:** `python -m app.core.seed` → calls `backend/app/core/seed.py:seed_database()`, which drops and recreates all tables, calls `populate(session)` from the generator, then immediately calls `run_detection(session)` to pre-generate alerts. The seed summary prints pool balances, transaction counts, and alert counts including how many anomaly labels are present server-side.

**Key invariant (tested):** `is_injected_anomaly` and `anomaly_type` fields are stored in the `transaction` table but are **never serialized into any API response**. All serializers (`TransactionOut`, etc.) omit these fields. Tests in the test suite assert that (a) the DB contains exactly 39 injected anomaly rows, and (b) no API response contains the words `is_injected_anomaly` or `anomaly_type` from these rows.

## Live Continuation: The Simulation Clock

When the demo clock runs (`POST /api/sim/start`), each tick calls `generate_tick(rng, plan)` in `backend/app/modules/sim/generator.py` with the clock's own RNG (`seed=7`, `SIM_SEED = 7` — independent of the seed generator's seed 1 so live traffic never collides with the seeded history).

Each tick (`SIM_MINUTES_PER_TICK = 5` minutes of synthetic time) generates transactions from the same direction-aware `build_pool_effects` logic, applies signed deltas to the live pool balances, then re-runs forecasts and incremental detection. The same `cash_out → physical_cash −, provider +` rule holds throughout.

**Baseline calm traffic per tick:** `BASELINE_COUNT = 3` transactions, `BASELINE_CASHOUT_SHARE = 0.35` (majority cash-in, gently refills physical cash). Kept below `velocity_min_count = 8` so calm traffic never self-triggers a velocity alert.

**Eid rush:** `EID_RUSH_TICKS = 6` ticks, `EID_INTENSITY_COUNT = {"low": 4, "medium": 8, "high": 12}`. At `intensity = "high"`, 12 cash-out transactions per tick (3,000–9,000 BDT each) drain physical cash rapidly and drive a fresh liquidity alert.

**Injected anomaly presets (live):**

| Type | Count | Amounts | Accounts |
|---|---|---|---|
| `structuring` | 8 | 4,910–4,990 BDT | ACC_7001, ACC_7002 |
| `velocity_spike` | 14 | 1,000–3,000 BDT | ACC_7101, ACC_7102 |
| `off_hours_burst` | 7 | 2,000–4,000 BDT | ACC_7201, ACC_7202 |

These use `7xxx` accounts to avoid overlapping the seeded `9xxx` anomaly clusters or the `1xxx` normal traffic pool.

## Assumptions

1. **One agent.** The generator is config-driven (`AGENTS` tuple in `synth/config.py`) and can extend to multiple agents, but only one is seeded.
2. **All transactions are `completed`.** No failed or pending transactions are generated; the analytics path does not model them.
3. **Integer BDT, no paisa.** All amounts are whole taka. Amounts are chosen to stay within Bangladesh Bank MFS limits for individual transactions.
4. **Fixed reference date.** No wall-clock reads occur in generation or tick math. The scenario is frozen at 2026-07-11, which is declared as an Eid-rush day.
5. **Static provider weights.** Cash-out volume is spread bKash 45%, Rocket 35%, Nagad 20% by fixed weights. Real agent traffic would vary by customer mix.
6. **Three anomaly types only.** Labeled ground truth covers structuring, velocity spike, and off-hours burst. Other named types (balance inconsistency) are not pre-injected in the seed but can be triggered live via `break_feed`.
7. **No cross-agent patterns.** Multi-agent coordination patterns (e.g., distributed structuring across several agent locations) are out of scope.

## Limitations

- **Single 3-hour scenario.** Metrics derived from this data measure coverage of known patterns, not discovery of novel ones.
- **Eid pattern is approximated.** The 82% cash-out rate during the Eid window is a plausible estimate, not a historical Bangladesh Bank observation.
- **No distribution shift.** Real traffic has drift, adversarial behavior (adaptations to detection), and data quality issues that a fixed synthetic scenario cannot represent.
- **Salary-day declared but not seeded.** The `salary_day` event window (07:00–09:00) is in the detector's event calendar but has no seeded transactions, so its false-positive suppression effect on salary traffic is not measured.
